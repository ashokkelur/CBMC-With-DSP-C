Contract verification

Most of the Ericsson code has contracts on function implementation. As we discussed in impl:contracts, we stubbed the precondition and postcondition with CBMC APIs. During Ericsson's code analysis we realised that contracts used in the code are run time guards, which is a way of pessimistically checking incoming parameters at run time. These conditions check the values of each parameter when a function is called and the return values are checked before returning. The contracts are not providing enough information to CBMC to verify the properties of each function.

In verification, contracts are expected to be defined on the behaviour of the function and also provide actual limits of the parameter. The preconditions for the function can include limits on parameters and limits on global states. The post conditions can include affected states and results being returned. The affected states can be either part of parameters or global variables.
